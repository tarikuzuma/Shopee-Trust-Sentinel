Shopee Trust Sentinel — Full Build Brief for Claude Code
What this is

A hackathon project (TRAILBLAZE Shopee AI Hackathon 2026, Fraud Detection track). We have ~1 day. This brief is a cold-start handoff — everything you need is here.

The problem, in one line

When a buyer files a return/refund claim on Shopee, they submit photo/video "proof." A huge share of that proof is faked. Right now a human reviews it case by case — slow, expensive, and Shopee's own regional team can't crack automated image-checking because of GPU cost and tiny privacy-limited datasets. We're building the autonomous system that judges whether return proof is authentic.

The exact task (scoped by the organizers)

Given a return case (stated return reason + image/video evidence), decide whether the proof is valid or invalid, autonomously, using front-end / face-value signals only. Backend data (IP, device, address, order history) is HARD out of scope — confidential, not available to us. Judge the evidence at face value.

The three ways proof gets faked (this is our detection target)

The training data labels every invalid case as one of:

Irrelevant proof — e.g. a video of the floor or an empty box that doesn't match the stated reason
Tampered — parcel already opened/swapped before filming; or no unboxing shown
AI-generated / edited — synthetic frames, splices, edited photos
Critical judge intel (this drives the design)
Cost asymmetry: a wrong "VALID" verdict hands money to fraudsters — that's the expensive mistake. A wrong "INVALID" is cheaper. So when uncertain, the system must lean toward reject/escalate, NEVER auto-approve. The judge said explicitly he'd rather see "escalate to human" than a confident wrong approval.
Automation is the goal. Escalate-to-human is an acceptable exception, not the default. Auto-decide as much as possible.
External VLMs (Claude/Gemini API) are explicitly allowed. Don't fine-tune our own model — no time, tiny dataset. Use an external vision model API.
Deliverables graded: (1) a working classifier, (2) a confusion matrix — TP/FP/TN/FN with an escalation count on the side, (3) a reproducible script/repo they can re-run at scale, (4) bonus: a sped-up demo video proving it runs fast (their real bottleneck is GPU cost/speed at Shopee scale).
Test data has no ground truth and is mixed valid/invalid. Training data is invalid-only (infer "valid" as absence of invalid signals). So our thought process/architecture is graded as much as raw accuracy.
Judges are cross-panel and mostly non-technical — clarity beats cleverness.
The architecture — "Rung 1", flat and autonomous (BUILD THIS)

Deliberately simple so it ships and demos clean. One case → four checks always run in parallel → scores combine → synthesis writes a brief → decision router acts. No dynamic routing, nothing fancy to break.

Flow:

Return case (reason + video + photo)
      ↓  (all run in parallel)
┌─────────────┬──────────────┬───────────┬────────────┐
│ Authenticity│ Completeness │  Tamper   │ Relevance  │   + Defender
│ AI-edited?  │ unboxing     │ pre-      │ matches    │   (argues the
│             │ shown?       │ opened?   │ reason?    │   claim is legit)
└─────────────┴──────────────┴───────────┴────────────┘
      ↓  each returns { score 0-1, reason_string, applicable: bool }
Scoring combiner → credibility 0–100 (higher = more trustworthy)
      ↓
Synthesis agent (LLM) → plain-language brief + verdict
      ↓
Decision router (asymmetric thresholds):
   credibility > ~75  → AUTO-APPROVE (valid, behind the scenes)
   credibility < ~35  → AUTO-REJECT  (invalid, behind the scenes)
   35–75 (uncertain)  → ESCALATE to human swipe queue
      ↓
Everything logged to SQLite → produces confusion matrix + labeled seed
The agents (each is a check that reads evidence and votes)
Authenticity — detects AI-editing/splices/synthetic frames. This is the hardest and highest-value one; the judge admitted his dev team can't crack it. Use an external VLM.
Completeness — does the video actually show the parcel being opened?
Tamper — was the parcel already open / seal broken before filming?
Relevance — does the evidence match the stated return reason? (floor video for a "broken item" claim = irrelevant)
Defender — the "not-guilty vote." Actively argues the claim is LEGITIMATE and pulls credibility UP. This protects honest buyers whose item really did arrive broken. It's what keeps us from being a blind auto-reject machine. It points OPPOSITE to the other four.
Scoring direction (lock this — do not mix)

Every signal outputs credibility (high = trustworthy), never risk (high = bad). Mixing directions makes the combined score cancel to mush. Low credibility escalates/rejects.

Critical implementation detail — the applicable flag

Every check returns applicable: true/false alongside its score. If a check can't run (e.g. Completeness needs video but the case is photo-only), it returns applicable: false and contributes NEUTRAL to the score — never suspicious. A missing signal must NOT lower credibility, or honest photo-only buyers get wrongly flagged. This is the single most important field to get right.

Fallback behavior
Check can't run (missing data) → returns not_applicable, neutral, others still vote.
All checks inconclusive / no information → do NOT guess, do NOT auto-approve → escalate to human. (No-information must never default to approve — cost asymmetry.)
Storage — SQLite (hybrid schema)

One .db file in the repo (reproducibility + zero setup + safe live demo). One table, cases:

Real columns (we filter/update on these): case_id, session_id, return_reason, credibility_score, decision (approve/reject/escalate), runtime_ms, created_at, human_verdict (filled later by swipe), true_label (filled if ground truth exists)
One TEXT column signals_json — the full per-signal breakdown (scores + reason strings + brief) stored as a JSON blob, since we only ever read it back whole, never query into it.

Why SQLite over flat JSON: our access pattern is read-filter-update (the swipe app updates a case after the human decides, and queries "escalated cases in this session"). That's SQL's home turf; flat files make in-place updates and filtered reads fiddly. But we keep the nested detail as JSON inside the row — best of both.

session_id groups a run so an analyst can return to a batch, see the stored remarks, and review why the AI decided what it did — without re-running.

The human escalation UI — the swipe app (BUILD, but scope tight)

A Tinder-style web app, correctly scoped as the exception path for low-confidence escalated cases ONLY. Each card shows: the credibility score, WHICH signal was ambiguous, and the synthesis brief. The safe gesture (hold/reject) is the easy swipe; approve is the deliberate one (cost asymmetry made physical). Every swipe writes human_verdict back to the DB → generates a labeled example → this IS the "seed for whatever comes next" the judge asked for. Do NOT build a separate admin/history browser — the swipe app filtered by session_id covers "look back."

The output artifact (this is graded — don't skip)

A script that queries the DB and prints/plots a confusion matrix: TP / FP / TN / FN, plus an escalation count column. Build the logging in from the start so this falls out for free.

Data contract (step 0 — build this FIRST, everything hangs off it)

Define one internal case-record shape (a dict/dataclass) with every field: case_id, session_id, return_reason, evidence_refs, video_frames, submitted_at, a per-agent output slot {signal_name, score, verdict, reason_string, confidence, applicable}, and final slots credibility_score, decision, brief, runtime_ms. Every component reads/writes this one shape. When real Shopee data arrives, we write ONE loader that maps their format into this shape — nothing else changes. The case-record shape IS the SQLite row.

Guardrail / thesis (don't let any "improvement" break this)

The AI investigates and decides autonomously for the confident cases, but the design must (a) always keep credibility direction consistent, (b) never auto-approve under uncertainty, (c) treat missing signals as neutral not suspicious, (d) keep the defender pulling honest claims up. The differentiator is: we separate "is the proof authentic" (the AI's job) from "whose fault is the damage" (a human's job, out of scope) — and we protect honest buyers, not just catch liars.

Both are stated in the brief; just make sure you two agree before building.
