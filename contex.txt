# Shopee Trust Sentinel — Full Build Brief for Claude Code (v2)

TRAILBLAZE Shopee AI Hackathon 2026, Fraud Detection track. ~1 day. v2 adds Rung 0, a deterministic pre-filter that runs before any paid model call.

## The problem, in one line

A buyer files a return/refund claim with photo/video proof. A share of that proof is faked. A human reviews every case by hand. We're building the system that judges proof validity autonomously, using front-end/face-value signals only — backend data (IP, device, address, order history) is out of scope.

## The exact task

Given a return case (stated reason + image/video evidence), decide valid or invalid, autonomously. Invalid cases fall into three types: irrelevant proof, tampered proof (parcel pre-opened, or no unboxing shown), AI-generated/edited proof.

## Judge intel that drives every design choice

- Cost asymmetry: a wrong VALID verdict pays a fraudster — expensive. A wrong INVALID is cheap. When uncertain, lean reject/escalate. Never auto-approve on uncertainty.
- Automation is the goal; escalate-to-human is the exception, not the default.
- External VLMs (Claude/Gemini API) are explicitly allowed. No fine-tuning — no time, no dataset for it.
- Graded deliverables: working classifier, confusion matrix (TP/FP/TN/FN + escalation count), reproducible script, bonus sped-up demo video.
- Training data is invalid-only; test data is unlabeled mixed valid/invalid. Architecture and reasoning are graded alongside accuracy.
- Judges are cross-panel, mostly non-technical. Clarity beats cleverness.

## Architecture — Rung 0 (new) + Rung 1

Rung 0 is a deterministic Python layer: `cv2` + `Pillow` + `imagehash`, zero tokens, zero API calls. It runs first and does two unrelated jobs that must not be filed under one name:

1. **Quality gating** — corrupt file, too blurry, too dark, too short. These are "can't be judged" outcomes. They route to **escalate** (`insufficient_evidence`), never reject. Auto-rejecting a blurry photo punishes an honest buyer with a cheap phone camera — the exact false positive the Defender exists to prevent, tripped one layer earlier. Low-end-device photos are a large slice of real traffic here, not an edge case.
2. **Duplicate detection** — perceptual hash (`imagehash.phash`) against every prior submitted proof. A reused image across unrelated claims is a genuine fraud signal, more reliable than anything the VLM agents produce, because it's deterministic. A match routes straight to **reject** (`duplicate_proof`), skipping the four agents entirely.

Same layer, two exit routes. A third check — EXIF read — never gates at all: it extracts the `Software` tag (Photoshop/GIMP/Lightroom/etc.) for free and hands it forward as a prior into the Authenticity agent's prompt, instead of making an expensive VLM call re-derive from pixels something a metadata read already answered.

Two boundary calls, decided and locked:

- **Frozen/looped frame** (a still image passed off as video) is NOT a Rung 0 quality signal. It's deception, not bad capture — it belongs to the **Tamper** agent. Rung 0 asks "can this be judged"; Tamper asks "does this look faked." A frozen frame can be judged, and what it shows is fraud, so it must be able to contribute to a reject verdict, not an escalate.
- **EXIF editing-software tag** never gets a verdict of its own. It's a prior, injected into the Authenticity agent's prompt.

Anything Rung 0 hard-stops (escalate or reject) never reaches a VLM — that's where the API cost savings come from, on top of catching a fraud class (duplicate proof) the four-agent layer doesn't otherwise cover.

```
Return case (reason + video + photo)
      ↓
┌─────────────────────────── Rung 0 — deterministic, zero tokens ───────────────────────────┐
│  integrity + quality (blur/dark/corrupt/too-short)   →  fail  →  ESCALATE insufficient_evidence
│  duplicate check (perceptual hash vs prior proofs)   →  fail  →  REJECT duplicate_proof
│  EXIF read (editing-software tag)                    →  always passes through as a prior
└──────────────────────────────────────────────────────────────────────────────────────────┘
      ↓ (pass-through only, + EXIF prior attached)
┌─────────────┬──────────────┬───────────┬────────────┐
│ Authenticity│ Completeness │  Tamper   │ Relevance  │   + Defender
│ AI-edited?  │ unboxing     │ pre-opened│ matches    │   (argues the
│ (+EXIF prior│ shown?       │ / frozen  │ reason?    │   claim is legit)
│  as context)│              │  frame?   │            │
└─────────────┴──────────────┴───────────┴────────────┘
      ↓  each returns { score 0-1, reason_string, applicable: bool }
Scoring combiner → credibility 0–100 (higher = more trustworthy)
      ↓
Synthesis agent (LLM) → plain-language brief + verdict
      ↓
Decision router (asymmetric thresholds):
   credibility > ~75  → AUTO-APPROVE (valid, behind the scenes)
   credibility < ~35  → AUTO-REJECT  (invalid, behind the scenes)
   35–75 (uncertain)  → ESCALATE to human swipe queue
      ↓
Everything logged to SQLite → confusion matrix + labeled seed
```

## Scoring direction (unchanged, still locked)

Every signal outputs credibility (high = trustworthy), never risk. Rung 0's two gates are the one exception to "everything feeds the combiner" — they act before the combiner exists for that case, by design, because they're deterministic enough to skip the deliberation step entirely.

## The applicable flag (unchanged, still the most important field)

Every check — Rung 0 included — returns `applicable: true/false`. A check that can't run contributes NEUTRAL, never suspicious. Rung 0's duplicate check is the clearest example: photo-only cases run it, video-only cases mark it not applicable and move on.

## Fallback behavior (unchanged)

No information anywhere in the pipeline → escalate, never auto-approve.

## Storage — SQLite, one new table

Existing `cases` table is untouched. Rung 0 adds:

```sql
CREATE TABLE IF NOT EXISTS proof_hashes (
    case_id TEXT PRIMARY KEY,
    phash   TEXT NOT NULL
);
```

Read-filter-update against this table is how the duplicate check works: every clean, non-duplicate case's hash gets written on the way through, so it becomes the comparison set for every later case. Kept as its own table, not folded into `cases`, since it's queried by hash value, not by the case-record fields the rest of the schema is built around.

## Data contract — one addition

Existing shape (`case_id`, `session_id`, `return_reason`, `evidence_refs`, `video_frames`, `submitted_at`, per-agent slots, `credibility_score`, `decision`, `brief`, `runtime_ms`) is unchanged. Add one slot:

- `rung0`: `{quality: {...}, duplicate: {...}, exif_prior: {...}}` — the three Rung 0 check outputs, same `{signal_name, applicable, verdict, reason_string, confidence}` shape as every other agent, for consistency and so the swipe app / confusion-matrix script can read it without a special case.
- `authenticity_prior`: the EXIF extract handed forward, read once by the Authenticity agent's prompt builder.

When a Rung 0 gate fires, `case["decision"]` and `case["gate_reason"]` are set immediately and the four-agent stage is skipped for that case — but the case still gets a full row in `cases`, so it still counts in the confusion matrix.

## Threshold discipline (the same rule extended)

Every Rung 0 threshold — resolution floor, blur/Laplacian floor, brightness range, video duration floor, duplicate-hash hamming distance — is a placeholder until tuned against the labeled dataset. Same rule as the 35/75 credibility split. Guess these and the system rejects or escalates a large share of honest low-end-device photos.

One technical detail that breaks silently if missed: the blur metric (variance of Laplacian) scales with image resolution. Resize to a fixed reference width before thresholding, or the same cutoff means something different for a 400px phone photo than a 4K one. For video, sample ~10 frames and take the median blur/brightness — one motion-blurred frame in an otherwise clear video should not fail the whole clip.

## Tech stack — one addition

- `opencv-python-headless`, `imagehash`, `Pillow` for Rung 0 — no network calls, no GPU, runs before any external API touches the case.
- Everything else unchanged: external VLM API (Claude/Gemini) for Authenticity + optionally Relevance, SQLite (`sqlite3`), plain HTML/JS swipe app.

## Build sequence (updated, dependency order)

1. Data contract + SQLite schema, including `proof_hashes` — unblocks everyone.
2. **Rung 0** — quality gate, duplicate hash, EXIF read. No external dependency, so it can be built and unit-tested in isolation before the agents exist. Gives a second demoable spine: "here's a case getting rejected/escalated before it ever costs a token."
3. Scoring combiner + decision router with hand-fed fake numbers.
4. The four checks + Defender (Relevance and Tamper first — cheapest; Authenticity is the hard VLM one, and now takes the EXIF prior as input; Tamper now also takes the frozen-frame motion score as input).
5. Synthesis agent (LLM brief — depends on all signals existing).
6. Confusion-matrix script (queries the DB; Rung 0 short-circuited cases must appear in it too).
7. Swipe app (parallelizable by the second partner).

## What to name but not build (unchanged, still roadmap)

Selective triage router, CS-authored triages in plain language, self-recalibration from swipe labels, anything using backend data.

## Guardrail / thesis — extended

The AI investigates and decides autonomously for confident cases, but the design must:

(a) always keep credibility direction consistent,
(b) never auto-approve under uncertainty,
(c) treat missing signals as neutral, not suspicious,
(d) keep the Defender pulling honest claims up,
(e) **never let a capture-quality problem auto-reject** — Rung 0's quality gate only escalates, and
(f) **never let a deterministic fraud signal wait on a VLM call** — Rung 0's duplicate check rejects directly, and the frozen-frame signal reaches Tamper as a first-class input rather than being silently absorbed into the quality gate.

The differentiator stays the same: "is the proof authentic" (the AI's job) is separated from "whose fault is the damage" (a human's job, out of scope). Rung 0 adds a second differentiator: an obvious, deterministic fraud check (duplicate proof) is caught before spending a single token, and an obvious non-fraud problem (bad camera) never gets treated as fraud.
